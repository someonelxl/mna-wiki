# Index 模板文档 v0.1

## 1. 文档定位
本文件用于定义本项目中 Index 页面的标准骨架。
它不负责解释项目为什么存在，也不负责定义对象结构本身，而是回答：
- 一个 Index 页面最少应包含哪些部分
- Case Index、Concept Index、Source Index 各自承担什么作用
- Index 页面应如何与对象页分工
- 人和 LLM 应如何围绕 Index 页面协同工作

本文件属于模板层文档，应受《项目总需求定位 v2.0》《系统设计文档 v1.0》《索引与入口骨架设计 v0.1》《Rulebook 初稿 v0.1》《Schema 初稿 v0.1》约束。

## 2. Index 模板的作用
Index 模板的作用有五个：
1. 为索引页提供统一骨架
2. 让入口层有稳定的最小结构
3. 让不同 Index 页面之间的职责边界清楚
4. 让用户和 LLM 能快速进入对象层
5. 为后续更复杂的导航、筛选和搜索预留基础

## 3. Index 页面在系统中的定位
Index 页面是 Wiki 层中的浏览入口页。
它不是对象页本身，也不是规则页，而是：
- 对象入口页
- 浏览组织页
- 轻量导航页
- 使用层辅助页

Index 页面不应承担：
- 重复对象全文
- 替代 Case / Concept / Source Record
- 定义对象结构
- 承载过重分析内容

## 4. Index 模板的设计原则
### 4.1 入口优先于内容堆积
Index 页的核心作用是帮助进入对象层，而不是把对象内容堆满一页。

### 4.2 轻量优先
当前阶段 Index 页应保持轻量，只承载最小识别信息与跳转入口。

### 4.3 分工清楚
Case Index、Concept Index、Source Index 应各自承担清楚职责，不互相替代。

### 4.4 不预设复杂交互
当前阶段 Index 页不应假设已经有：
- 搜索系统
- 自动筛选器
- 动态聚类
- 可视化图谱
因此模板必须能在纯文档条件下成立。

### 4.5 AI 可协助整理，但不应随意重构入口体系
AI 可以帮助整理索引页首版，但入口结构和分工边界应由规则文档与设计文档约束。

## 5. 当前阶段 Index 页面的通用骨架
当前阶段，所有 Index 页面应至少包含以下 6 个部分。

## 5.1 页面基本信息
### 作用
说明这个 Index 页是什么、服务什么对象。
### 建议承载的信息
- Index 名称
- 服务对象
- 当前用途
- 当前阶段状态

### 原则
这一部分要让人一眼知道：
- 这页是干什么的
- 该从这里进入什么对象

---

## 5.2 页面职责说明
### 作用
明确这页负责什么，不负责什么。
### 原则
这一部分非常重要，因为 Index 页面最容易越权。
应明确说明：
- 它是入口页，不是对象替代页
- 它只提供最小导航和识别
- 详细内容应进入对象页查看

---

## 5.3 当前收录范围
### 作用
说明这页当前索引哪些对象。
### 原则
这部分应回答：
- 当前纳入了哪些对象
- 纳入标准是什么
- 当前是否是“正式对象索引”还是“临时收录”

### 注意
当前阶段不要求写成复杂纳入规则，但要让范围感清楚。

---

## 5.4 列表主体
### 作用
这是 Index 页最核心的部分。
用于列出对象入口。

### 原则
列表主体应只放：
- 最小识别信息
- 必要的分类信息
- 对应对象页入口

### 注意
这里不应复制对象正文，不应变成超长摘要页。

---

## 5.5 使用说明 / 浏览建议
### 作用
告诉用户怎么使用这页。
### 原则
这一部分当前阶段保持轻量即可。
例如可说明：
- 先看编号
- 再看名称
- 再跳转对象页
- 详细分析请进入对象页

---

## 5.6 待扩展区
### 作用
为未来扩展预留位置。
### 可承载内容
- 后续可加的分类方式
- 后续可加的筛选维度
- 后续待补对象
- 结构升级方向

### 原则
这部分不宜写太重，但应保留，因为入口层天然会继续扩展。

## 6. Case Index 的专用骨架
Case Index 是当前阶段最重要的 Index 页面。

### 6.1 页面职责
Case Index 应承担：
- 作为 Case 的主入口
- 提供最基础的案例浏览入口
- 帮助快速识别案例并进入单案例页
- 为后续按交易结构、行业、地区扩展预留骨架

### 6.2 最小列表字段
Case Index 列表主体当前阶段建议至少包括：
- `case_id`
- `case_name`
- `transaction_structure`
- `industry`
- `region`
- `status`
- 可选的一句话定义

### 6.3 当前阶段不要求
- 复杂筛选
- 自动排序逻辑
- 全量比较视图
- 复杂标签体系

## 7. Concept Index 的专用骨架
Concept Index 是当前阶段第二重要的 Index 页面。

### 7.1 页面职责
Concept Index 应承担：
- 作为 Concept 的主入口
- 帮助从母题进入案例网络
- 帮助浏览当前已正式纳入的母题
- 为后续按母题类型、启用状态、优先级扩展预留骨架

### 7.2 最小列表字段
Concept Index 列表主体当前阶段建议至少包括：
- `concept_id`
- `concept_name`
- `concept_type`
- `status`
- 可选的一句话核心逻辑

### 7.3 当前阶段不要求
- 复杂母题树
- 多层级知识图谱
- 自动关系展开

## 8. Source Index 的专用骨架
Source Index 是当前阶段的轻量辅助入口。
它是全局来源入口，不是单案例来源精细管理页；单案例来源复杂度应优先回到 Case 页面内部处理。

### 8.1 页面职责
Source Index 应承担：
- 作为 Source Record 的轻量入口
- 帮助回溯来源记录
- 帮助从来源进入相关 Case / Concept
- 为后续来源池管理预留基础

### 8.2 最小列表字段
Source Index 列表主体当前阶段建议至少包括：
- `source_id`
- `title`
- `source_type`
- `author_or_publisher`
- `published_at`
- 可选的主要关联对象

### 8.3 当前阶段不要求
- 全量来源管理系统
- 来源评分
- Raw Source 浏览主入口
- 复杂来源聚类

## 9. 三类 Index 页面之间的分工
### 9.1 Case Index
主入口，服务单案例进入与案例浏览。

### 9.2 Concept Index
第二入口，服务母题进入与跨案例组织。

### 9.3 Source Index
辅助入口，服务来源回溯与研究支撑。

### 9.4 当前阶段的总原则
三者并列存在，但重要性不同：
- Case Index 最重要
- Concept Index 次之
- Source Index 保持轻量

## 10. Index 页面与对象页的关系
### 10.1 Index 页不替代对象页
Index 页只提供最小入口和识别信息。
详细内容应进入：
- Case 页面
- Concept 页面
- Source Record 页面

### 10.2 Index 页可引用对象最小字段
Index 页可以引用对象的最小识别字段，但不应复制分析正文。

### 10.3 Index 页与对象页应通过稳定入口连接
当前阶段应优先依赖：
- 对象编号
- 对象名称
- 页面链接关系
来组织入口。

## 11. AI 与人的协作方式
### 11.1 AI 负责什么
AI 在 Index 页面生成中，优先负责：
- 整理首版入口结构
- 汇总对象最小识别字段
- 建议列表分组方式
- 检查索引页是否过重或越权

### 11.2 人负责什么
人优先负责：
- 决定索引页职责边界
- 决定哪些对象正式纳入索引
- 判断哪些字段适合留在索引层
- 决定是否需要扩展维度

### 11.3 当前阶段的默认原则
AI 优先整理首版，人的结构判断优先于 AI 的自由扩展。

## 12. 当前阶段的最小完成标准
一个 Index 页面在当前阶段，如果满足以下条件，可视为达到“最小可用”状态：
1. 页面职责清楚
2. 当前收录范围清楚
3. 列表主体存在
4. 列表字段足够识别对象
5. 用户能从 Index 页进入对象页
6. 页面没有越权变成对象替代页

## 13. 本模板文档的作用
本文件用于为后续：
- Case Index 创建
- Concept Index 创建
- Source Index 创建
- Codex 生成索引页首版
- review / lint 检查入口层

提供统一骨架。
未来如果 Index 页面结构需要升级，应优先更新本文件，而不是让各个 Index 页面自由长成不同风格。

## 14. 当前结论
当前阶段的 Index 模板核心可以概括为：
- 先把入口职责讲清楚
- 再把收录范围讲清楚
- 再把对象列表放进去
- 再为未来扩展预留轻量空间

只要这条骨架立住，系统就已经从“有对象”进入“对象可被组织和进入”的状态。
