# standard-writeback-v0-1

## 1. 文档定位
本文件用于定义本项目当前阶段的 write-back 最小执行标准。
它承接《项目总需求定位 v2.0》《系统设计文档 v1.0》《Rulebook 初稿 v0.1》《Schema 初稿 v0.1》以及三份模板文档。
本文件不负责定义对象结构本身，也不负责定义 ingest 输入处理，而是回答：
- 新的理解、判断和连接在什么条件下值得写回
- 应优先写回到哪个对象
- 当前阶段 write-back 的边界、顺序和风险控制是什么

## 2. write-back 的定义
在本项目中，write-back 指把新的理解、判断和连接正式写回系统的知识层。
它不是把原始来源重复存档，也不是把一次对话临时想法直接塞进页面，而是：
1. 从候选更新中筛出值得长期保留的内容
2. 将其写回适当的 wiki 对象
3. 让系统通过使用逐步增长

因此，write-back 的重点不是“记录所有想法”，而是“沉淀值得保留的结构化知识”。

## 3. 当前阶段 write-back 的适用输入
当前阶段 write-back 主要面向以下四类输入：

### 3.1 案例判断
例如：
- 某案例更准确的一句话定义
- 某案例更稳的交易逻辑分析
- 某案例新的核心冲突提炼

### 3.2 母题判断
例如：
- 某 Concept 的核心逻辑更清晰
- 某 Concept 的适用边界更准确
- 某 Concept 的常见误判被更好地定义

### 3.3 连接关系
例如：
- 新的 Case → Concept 连接
- 新的 Case → Case 连接
- 新的 Source → Case / Concept 连接

### 3.4 来源价值提炼
例如：
- 某来源为什么值得保留
- 某来源支持了哪个关键判断
- 某来源应优先服务哪个对象

## 4. 当前阶段 write-back 的原则
### 4.1 候选优先于直接写回
当前阶段默认原则是：
**先形成候选更新，再决定是否正式写回。**

### 4.2 写回优先进入知识层，不污染来源层
write-back 的正式目标是 wiki 层对象。
原则上不应把总结性内容写回 raw source 层。
原件层 / 原始出处层不是 write-back 的目标层，只作为证据链来源存在。

### 4.3 写回优先服务系统稳定增长
只有那些能提升：
- 对象清晰度
- 连接质量
- 内容复用价值
- 分析稳定性
的内容，才值得正式写回。

### 4.4 未成熟想法优先停留在候选层
当前阶段不追求“把所有想法都写回”，而追求“把经得住复用的想法写回”。

## 5. 当前阶段 write-back 的目标位置
当前阶段 write-back 优先写回以下对象：

### 5.1 Case 页面
正式落点：
- `wiki/cases/`

适合写回的内容包括：
- 一句话定义
- 交易逻辑分析
- 核心冲突
- 相关 Concept
- 相关案例
- 来源引用入口

### 5.2 Concept 页面
正式落点：
- `wiki/concepts/`

适合写回的内容包括：
- 核心逻辑
- 适用边界
- 常见误判
- 代表案例
- 相关来源
- 可讲角度

### 5.3 Source Record 页面
正式落点：
- `wiki/sources/`

适合写回的内容包括：
- 关键来源性质说明
- 核心要点
- 关联对象
- 后续动作建议
- 风险提示

### 5.4 当前阶段暂缓写回的位置
当前阶段原则上暂缓大规模写回：
- `wiki/analysis/`
- `wiki/indexes/`
除非目标非常明确、内容已经足够稳定。

## 6. 哪类内容值得 write-back
当前阶段优先写回以下内容：

### 6.1 结构化且可复用的判断
例如：
- 某案例最准确的一句话定义
- 某交易结构真正的分析重点
- 某 Concept 最稳定的核心逻辑

### 6.2 有来源支撑的分析结论
如果某条判断有明确来源支持，且足够稳定，优先考虑写回。

### 6.3 能提升连接质量的关系
例如：
- 某案例应正式挂接到某母题
- 某来源应正式归入某来源池
- 某案例和另一案例之间存在值得长期保留的比较关系

### 6.4 对内容生产有明显价值的结构化提炼
例如：
- 某案例最值得讲的核心冲突
- 某母题最适合组织的案例群
- 某来源最值得提炼的认知矛盾

## 7. 哪类内容当前不宜 write-back
当前阶段不应轻易写回以下内容：

### 7.1 没有来源支撑的判断
如果只是对话中一闪而过的想法，没有证据层支撑，默认不写回正式知识层。

### 7.2 情绪性表达
例如：
- 口语化临时评价
- 明显偏情绪的措辞
- 还未经整理的结论句

### 7.3 过于碎片化的点子
如果一条内容还不能形成稳定的对象字段或连接价值，应优先停留在候选层。

### 7.4 会污染 raw source 的内容
总结、判断、分析结论不应写回 raw source 正文。

## 8. 当前阶段 write-back 的基本顺序
### 第一步：识别内容类型
先判断当前候选内容属于：
- 案例判断
- 母题判断
- 连接关系
- 来源提炼

### 第二步：识别写回对象
判断它应写回：
- Case
- Concept
- Source Record
- 或暂不正式写回

### 第三步：判断是否满足写回条件
至少判断：
- 是否有来源支撑
- 是否已足够稳定
- 是否确实提升系统价值

### 第四步：决定写回级别
当前阶段优先分成两种：
- 候选更新
- 正式写回

### 第五步：正式写回时保持最小必要修改
正式写回应优先：
- 精确
- 最小
- 定位明确
而不是大段重写整个页面。

## 9. 当前阶段的正式写回条件
当前阶段，内容满足以下条件时，才适合正式写回：

### 条件一：对象归属清楚
明确知道这条内容该进入哪个对象。

### 条件二：来源支撑清楚
能够明确说出它由哪些 source 或 raw source 支撑。

### 条件三：表达已经过整理
不是临时对话碎片，而是已经变成可复用的结构化表达。

### 条件四：写回能提升系统质量
而不是只是把页面变长。

至少满足以上四项中的前三项，才建议正式写回。

## 10. 当前阶段不允许的 write-back 行为
### 10.1 不允许把对话草稿直接大段塞进正式页面
### 10.2 不允许在无审核下批量写正式知识页
### 10.3 不允许把总结性内容写回 raw source
### 10.4 不允许为了“留痕”而硬写回低价值内容
### 10.5 不允许把非标准交易自由简化成 buyer / seller 语义再写回 Case

## 11. 人与 Codex 的分工
### 11.1 Codex 可以做的 write-back 工作
当前阶段 Codex 可以：
- 整理候选更新
- 建议写回位置
- 协助把候选内容重写成结构化表达
- 执行小范围、明确授权的正式写回
- 提醒哪些内容还不适合写回

### 11.2 Codex 不应直接做的 write-back 工作
当前阶段 Codex 不应：
- 在无审核下批量更新正式页
- 自由扩写整页内容
- 自由把暂不稳定的分析当成最终结论
- 自由决定复杂案例的最终表述方式

### 11.3 人负责的关键动作
当前阶段人应优先负责：
- 判断是否值得正式写回
- 判断写回对象是否正确
- 判断措辞是否足够稳定
- 审核关键结论和关键连接

## 12. 当前阶段 write-back 的最小记录要求
任何一次正式 write-back，至少应能回答以下问题：
1. 写回的内容是什么
2. 写回到哪个对象
3. 依据哪些来源或候选更新
4. 为什么值得正式沉淀
5. 是否会影响其他对象的连接或理解

如果这 5 个问题答不清，就不应视为一次合格 write-back。

## 13. 当前阶段的最小完成标准
一次 write-back 在当前阶段，如果满足以下条件，可视为达到最小标准：
- 写回对象明确
- 内容已结构化
- 来源支撑明确
- 不污染 raw source
- 不越过审核边界
- 写回能提升对象清晰度、连接质量或内容复用价值

## 14. 本文档的作用
本文件用于为后续：
- Case / Concept / Source Record 的正式更新
- 候选更新转正式写回
- Codex 协助写回
- review 后的修正写回

提供最小执行标准。
后续如果 write-back 流程变得更复杂，应在本文件基础上扩展版本，而不是绕开本标准自由实现。

## 15. 当前结论
当前阶段的 write-back 标准可以概括为：
- 先候选
- 后写回
- 先判断值不值得
- 再判断写到哪里
- 只写回稳定、可复用、有来源支撑的内容

只要这套最小纪律立住，系统就能通过使用逐步长出真正有价值的知识，而不是长出噪音。
