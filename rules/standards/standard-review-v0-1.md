# standard-review-v0-1

## 1. 文档定位
本文件用于定义本项目当前阶段的 review 最小执行标准。
它承接《项目总需求定位 v2.0》《系统设计文档 v1.0》《Rulebook 初稿 v0.1》《Schema 初稿 v0.1》以及模板、命名实施细则和 ingest / write-back 标准。
本文件不负责定义对象结构本身，也不负责定义对象如何进入系统，而是回答：
- 当前阶段应检查什么
- 哪些问题属于必须处理的问题
- 哪些问题只需记录为风险
- review 的最小顺序、边界和输出要求是什么

## 2. review 的定义
在本项目中，review 指对当前系统状态进行结构性检查与质量核对。
它不是新内容生产，也不是自动化清洗，而是：
1. 检查对象、目录、命名和连接是否符合规则
2. 检查知识层与来源层是否发生混写
3. 检查模板、Schema、Rulebook、标准文档是否彼此打架
4. 输出问题清单、风险提示和最小修正建议

review 不负责生成候选更新，也不直接执行正式写回。

因此，review 的重点不是“继续写”，而是“确认现在有没有长歪”。

## 3. 当前阶段 review 的适用范围
当前阶段 review 主要面向以下六类对象或文档：

### 3.1 目录结构
例如：
- `docs/`
- `rules/`
- `templates/`
- `raw-sources/`
- `wiki/`
- `scripts/`

### 3.2 核心对象页面
例如：
- Case
- Concept
- Source Record
- Raw Source

### 3.3 规则文档
例如：
- Rulebook
- Schema
- 对象命名实施细则
- ingest / write-back / review 标准

### 3.4 模板文档
例如：
- Case 模板
- Concept 模板
- Source / Raw Source 模板

### 3.5 来源与知识层关系
例如：
- raw source 是否保持原始性
- Source Record 是否回得到原始出处
- Case / Concept 是否正确引用来源

### 3.6 Git 与项目状态
例如：
- 是否仍有未收敛中间态
- 是否存在路径漂移
- 是否有关键文件未进入版本管理

## 4. 当前阶段 review 的目标
当前阶段 review 主要有五个目标：

1. 检查对象边界是否清楚
2. 检查目录归属是否正确
3. 检查命名、字段和模板是否一致
4. 检查来源层与知识层是否被污染
5. 检查当前状态是否足以继续下一轮施工

## 5. 当前阶段 review 的重点检查项
当前阶段优先检查以下问题。

## 5.1 结构错位
例如：
- 文件放错目录
- 对象归属和目录归属不一致
- 模板已经定义了对象，但系统没有对应目录
- 新现实结构与 README / 设计文档不一致

## 5.2 对象边界漂移
例如：
- Case 被写成来源容器
- Concept 被写成普通标签页
- Source Record 被当成 raw source 正文
- Raw Source 被写成总结页

## 5.3 命名漂移
例如：
- 文件系统层没有使用 `kebab-case`
- 对象编号不稳定
- 标题和文件名关系混乱
- 交易结构受控类型被自由写法替代

## 5.4 规则不一致
例如：
- Rulebook 与 Schema 打架
- Schema 与模板不一致
- 命名实施细则与模板落地冲突
- 标准文档和实际操作边界不一致

## 5.5 来源支撑不足
例如：
- 重要判断没有来源入口
- Source Record 不能回到原始出处
- Raw Source 不能回到原件层
- 写回内容没有证据支撑

## 5.6 连接缺口
例如：
- Case 没有关联 Concept
- Concept 没有代表案例
- Source Record 没有关联对象
- 应该存在的 cross reference 尚未建立

## 5.7 Git 中间态
例如：
- 有重要路径变动未收敛
- 关键文件未进入版本管理
- 工作区不干净但仍继续施工
- README / 规则 / 模板修改后未同步提交

## 6. 当前阶段不把什么当成 review 的优先事项
为了保持 review 克制，当前阶段不优先检查以下内容：
- 文风是否漂亮
- 文案是否适合直接发布
- 页面是否已经写满
- 自动化效率是否最优
- 所有案例是否都已完整录入
- 所有母题是否已充分展开

当前阶段 review 先保秩序，不保丰满。

## 7. 当前阶段 review 的基本顺序
当前阶段 review 应遵循以下最小顺序。

### 第一步：先看结构
先检查：
- 目录是否对
- 文件是否放对层
- 对象是否放对目录

### 第二步：再看规则一致性
再检查：
- Rulebook
- Schema
- 标准文档
- 模板
之间是否一致

### 第三步：再看对象质量
再检查：
- Case
- Concept
- Source Record
- Raw Source
是否符合当前最小要求

### 第四步：再看连接与来源
再检查：
- 来源是否可追溯
- 连接是否成立
- 关键对象是否缺少挂接

### 第五步：最后判断是否可继续施工
review 的结论不是“完美”，而是：
- 是否可以安全进入下一轮施工
- 是否必须先停下来修问题

## 8. 当前阶段 review 的输出级别
当前阶段 review 的输出建议分为三类：

### 8.1 正常项
已经正确、无需处理的关键部分。

### 8.2 问题项
真实存在、应明确指出的问题。

### 8.3 风险项
当前不一定构成错误，但后续很容易出问题的地方。

## 9. 当前阶段哪些问题必须先修
以下问题在当前阶段应视为高优先级修复项：

### 9.1 路径错位
如果文件或对象放错目录，应优先修正。

### 9.2 规则冲突
如果 Rulebook、Schema、模板彼此打架，应优先收敛。

### 9.3 来源层污染
如果 raw source 被写成总结页，应优先纠正。

### 9.4 命名规则被破坏
如果对象编号、文件名、受控类型明显漂移，应优先收敛。

### 9.5 Git 状态未收敛
如果项目仍处于重要中间态，不应继续下一轮正式施工。

## 10. 当前阶段哪些问题可以暂缓
以下问题当前阶段可以记录，但不必立刻停工处理：

### 10.1 模板中的自然语言栏目名与 Schema 字段名未完全一一映射
只要对象逻辑一致，可暂时接受。

### 10.2 个别页面尚未写满全部字段
只要达到最小可用状态，可暂时接受。

### 10.3 某些索引或 analysis 页面尚未建立
当前阶段不是必须项。

### 10.4 某些风险尚未完全显式写入文档
只要不构成当前施工冲突，可先记录为风险项。

## 11. 人与 Codex 的分工
### 11.1 Codex 可以做的 review 工作
当前阶段 Codex 可以：
- 检查目录结构
- 核对文档路径
- 对比 Rulebook / Schema / 模板的一致性
- 发现旧路径残留
- 列出问题项与风险项
- 给出最小修正建议

### 11.2 Codex 不应直接做的 review 工作
当前阶段 Codex 不应：
- 在 review 过程中顺手大改一堆文件
- 自由扩大检查范围
- 生成候选更新或直接执行正式写回
- 把风险项当成必须修复项强行处理
- 在未经确认的情况下自行改变对象边界或系统结构

### 11.3 人负责的关键动作
当前阶段人应优先负责：
- 判断问题优先级
- 判断哪些问题必须先修
- 决定是否暂停施工
- 决定下一轮施工任务

## 12. 当前阶段 review 的最小记录要求
任何一次正式 review，至少应能回答以下问题：
1. 本次 review 检查了哪些范围
2. 发现了哪些真实问题
3. 哪些是风险，不是错误
4. 当前是否可继续施工
5. 下一步最小修正建议是什么

如果这 5 个问题答不清，就不应视为一次合格 review。

## 13. 当前阶段的最小完成标准
一次 review 在当前阶段，如果满足以下条件，可视为达到最小标准：
- 检查范围明确
- 结构问题已识别
- 规则一致性已核对
- 来源层与知识层边界已检查
- 输出中区分了正常项、问题项和风险项
- 对“是否可继续施工”给出了明确判断

## 14. 本文档的作用
本文件用于为后续：
- 路径收敛检查
- 六文档一致性体检
- 模板与 Schema 对齐检查
- Git 中间态检查
- 下一轮施工前的健康检查

提供最小执行标准。
后续如果 review 机制变得更复杂，应在本文件基础上扩展版本，而不是绕开本标准自由检查。

## 15. 当前结论
当前阶段的 review 标准可以概括为：
- 先看结构
- 再看规则
- 再看对象
- 再看连接
- 最后判断能不能继续施工

只要这套最小纪律立住，项目就能在持续建设中不断体检，而不是等到长歪后再被动返工。
