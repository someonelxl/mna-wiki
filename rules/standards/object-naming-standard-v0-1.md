# 对象命名实施细则 v0.1

## 1. 文档定位
本文件用于承接《命名规范总纲 v1.0》《Rulebook 初稿 v0.1》《Schema 初稿 v0.1》，把“命名原则”进一步下沉为“对象级命名实施规则”。
本文件不负责解释项目为什么存在，也不负责定义对象字段结构，而是回答：
- Case、Concept、Source Record、Raw Source 应如何编号
- 文件名应如何与对象编号配合
- 页面标题与文件名应如何区分
- 后续新增对象时应如何保持命名稳定

## 2. 文档作用
本文件的作用有四个：
1. 固定核心对象的编号体系
2. 固定对象文件名的构成方式
3. 固定“显示标题”和“文件名”的分层规则
4. 降低人和 LLM 在后续建页时的命名漂移风险

## 3. 当前阶段纳入命名细则的对象
当前阶段只纳入以下四类对象：
- Case
- Concept
- Source Record
- Raw Source

当前阶段暂不纳入：
- Index
- Analysis
- Summary
- Decision Log
- Review Report

这些对象后续如进入正式建设，再补独立命名规则。

## 4. 命名实施的总原则
### 4.1 对象必须有稳定编号
所有核心对象必须有稳定编号，不应只靠自然语言标题识别。

### 4.2 文件名服务于系统，标题服务于阅读
文件名应工程化、稳定、便于 Git / Terminal / Codex 使用。
页面标题应人类可读、清楚、克制。

### 4.3 编号一旦建立，不轻易重排
对象编号的作用是稳定引用，不是临时排序。
对象新增时应向后续号，而不是为追求整齐频繁改旧编号。

### 4.4 前缀应表达对象类别
编号前缀应明确表达对象所属类型，避免跨对象混用。

### 4.5 文件名不重复写过多语义
对象文件名不应把全部说明句塞进去，而应以“编号 + 简洁 slug”为主。

## 5. 编号体系总览
当前阶段正式采用如下编号前缀：

- Case：`CASE-`
- Concept：`CONCEPT-`
- Source Record：`SRC-`
- Raw Source：`RAW-`

## 6. 位数规则
当前阶段统一使用三位数字编号：
- `001`
- `002`
- `003`

因此对象编号格式应为：
- `CASE-001`
- `CONCEPT-001`
- `SRC-001`
- `RAW-001`

### 说明
三位数字足以覆盖当前阶段，又保留了后续扩展空间。
后续若数量显著增加，可扩展到四位，但不得随意回改旧编号。

## 7. Case 命名规则
### 7.1 编号格式
Case 编号统一使用：
`CASE-001`

### 7.2 文件名格式
Case 文件名统一使用：
`case-001-<slug>.md`

### 7.3 slug 规则
`<slug>` 应为案例的简洁英文或拼音式标识，使用 `kebab-case`。
要求：
- 小写
- 单词之间用 `-`
- 尽量简短
- 优先体现案例最核心可识别主体
- 不把过长说明写进 slug

### 7.4 示例
- `case-001-prada-versace.md`
- `case-002-openai-tbpn.md`
- `case-003-bytedance-moonton.md`

### 7.5 页面标题规则
页面标题可采用自然语言，例如：
- `CASE-001｜Prada 收购 Versace`
- `CASE-002｜OpenAI 收购 TBPN`
- `CASE-003｜字节跳动收购沐瞳`

### 7.6 标题与文件名关系
- 文件名：工程化、便于系统使用
- 标题：面向人阅读
两者不要求完全一致，但必须能相互对应。

## 8. Concept 命名规则
### 8.1 编号格式
Concept 编号统一使用：
`CONCEPT-001`

### 8.2 文件名格式
Concept 文件名统一使用：
`concept-001-<slug>.md`

### 8.3 slug 规则
`<slug>` 应为母题的简洁英文标识，使用 `kebab-case`。
要求：
- 小写
- 单词之间用 `-`
- 尽量表达母题核心逻辑
- 不写成长句
- 不用临时缩写，除非已成为稳定术语

### 8.4 示例
- `concept-001-control-shift.md`
- `concept-002-reverse-merger.md`
- `concept-003-spin-off-repricing.md`

### 8.5 页面标题规则
页面标题可采用自然语言，例如：
- `CONCEPT-001｜控制权转移`
- `CONCEPT-002｜借壳上市`
- `CONCEPT-003｜分拆重估`

## 9. Source Record 命名规则
### 9.1 编号格式
Source Record 编号统一使用：
`SRC-001`

### 9.2 文件名格式
Source Record 文件名统一使用：
`src-001-<slug>.md`

### 9.3 slug 规则
`<slug>` 应优先体现来源主题或出处主体，使用 `kebab-case`。
要求：
- 小写
- 单词之间用 `-`
- 尽量简洁
- 重点体现“这是什么来源”

### 9.4 示例
- `src-001-prada-annual-report-2024.md`
- `src-002-openai-tbpn-announcement.md`
- `src-003-moonton-deal-report.md`

### 9.5 页面标题规则
页面标题可采用自然语言，例如：
- `SRC-001｜Prada 2024 年报`
- `SRC-002｜OpenAI 关于 TBPN 的公告`
- `SRC-003｜Moonton 交易分析报告`

## 10. Raw Source 命名规则
### 10.1 编号格式
Raw Source 编号统一使用：
`RAW-001`

### 10.2 文件名格式
Raw Source 文件名统一使用：
`raw-001-<slug>.md`

### 10.3 slug 规则
`<slug>` 应优先体现原始资料主题或出处主体，使用 `kebab-case`。

### 10.4 示例
- `raw-001-prada-annual-report-2024.md`
- `raw-002-openai-tbpn-web-clip.md`
- `raw-003-moonton-video-transcript.md`

### 10.5 页面标题规则
页面标题可采用自然语言，例如：
- `RAW-001｜Prada 2024 年报文本层`
- `RAW-002｜OpenAI 收购 TBPN 网页剪藏`
- `RAW-003｜Moonton 视频转录文本`

## 11. 对象编号与文件名的配合规则
### 11.1 编号必须进入文件名
当前阶段，核心对象文件名必须显式包含对象编号。
不能只用 slug 作为唯一识别方式。

### 11.2 文件名编号部分统一小写对象类型
虽然对象正式编号是：
- `CASE-001`
- `CONCEPT-001`
- `SRC-001`
- `RAW-001`

但文件名中统一采用小写类型前缀：
- `case-001-...`
- `concept-001-...`
- `src-001-...`
- `raw-001-...`

这是为了与文件系统层的 `kebab-case` 保持一致。

### 11.3 页面标题中保留正式编号格式
页面标题里可以保留大写正式编号：
- `CASE-001｜...`
- `CONCEPT-001｜...`

## 12. 编号生成规则
### 12.1 新对象默认顺序递增
新增对象时，应在同类对象中顺序递增：
- 已有 `CASE-001`、`CASE-002`
- 下一个就是 `CASE-003`

### 12.2 不因删除或废弃回补旧编号
如果某个对象废弃，不应为了“整齐”把后续编号整体重排。
旧编号可以空出，但不应重用。

### 12.3 同一对象编号与文件名必须唯一
一个编号只能对应一个正式对象页面。
不得让两个不同页面共用同一编号。

## 13. slug 使用规则
### 13.1 slug 是辅助识别，不是主身份
对象真正的稳定身份是编号，不是 slug。
slug 的作用是帮助人快速识别文件主题。

### 13.2 slug 可以在严格控制下调整
如果对象理解发生变化，slug 可以在必要时微调，但：
- 编号不变
- 调整应有明确理由
- 不应频繁改动

### 13.3 slug 不应包含这些内容
- `final`
- `new`
- `latest`
- `temp`
- `test`
- 日期后缀（除非日期本身是对象本体的一部分）
- 过长的说明句

## 14. 标题规则
### 14.1 标题应优先服务阅读
标题允许中文，允许自然语言。
标题不要求工程化，但要：
- 清楚
- 简洁
- 不口号化
- 不夸张

### 14.2 标题建议保留对象编号
为了提高检索和人工核对效率，标题建议采用：
`对象编号｜自然语言标题`

### 14.3 标题与文件名可以不同层表达
例如：
- 文件名：`case-003-bytedance-moonton.md`
- 页面标题：`CASE-003｜字节跳动收购沐瞳`

这是允许且推荐的。

## 16. 本文档的作用
本文件的作用，是在当前阶段先把四类核心对象的命名实施规则定住。
以后：
- 新建正式对象页
- Codex 生成新页面
- review / lint 检查命名
- 未来做轻量自动化编号
都应以本文件为准。

## 17. 当前结论
当前阶段的对象命名实施规则可以概括为：
- 对象用正式编号保证稳定身份
- 文件名用 `kebab-case` 保证系统稳定
- 标题用自然语言保证阅读清晰
- 编号不轻易改，slug 不自由漂

只要这套规则先定住，后面正式建页时，命名就不会快速失控。
