# standard-ingest-v0-1

## 1. 文档定位
本文件用于定义本项目当前阶段的 ingest 最小执行标准。
它承接《项目总需求定位 v2.0》《系统设计文档 v1.0》《Rulebook 初稿 v0.1》《Schema 初稿 v0.1》以及三份模板文档。
本文件不负责定义对象结构本身，而是回答：
- 新案例、新来源、新分析结果进入系统时，最小应如何处理
- 什么时候只形成候选更新，什么时候可以正式写入
- 当前阶段 ingest 的边界、顺序和输出要求是什么

## 2. ingest 的定义
在本项目中，ingest 不是简单导入文件，而是：
1. 接收新的输入
2. 判断其对象类型与影响范围
3. 将其放入正确层级
4. 生成候选更新或正式记录
5. 为后续 write-back、review 和内容生产提供基础

因此，ingest 的重点不是“把东西放进项目”，而是“让系统开始正确吸收它”。

## 3. 当前阶段 ingest 的适用输入
当前阶段 ingest 主要面向以下三类输入：

### 3.1 新案例
例如：
- 一个新的并购案例名称
- 一个新的资本运作事件
- 一组已知案例基础事实

### 3.2 新来源
例如：
- 网页链接
- PDF 原件
- 视频链接
- 书籍、研报、文献、报告
- 从原件转出的文本层

### 3.3 新分析结果
例如：
- 对某案例的新判断
- 对某母题的新解释
- 对多个案例的新连接
- 对来源价值的新归纳

## 4. ingest 的层级判断规则
任何新输入进入系统前，必须先判断它属于哪一层。

### 4.1 原件层
原始 PDF、原始网页链接、原始视频链接、原始文件位置，属于原件层或原始出处层。
它们不是 raw source 本体，但应被记录为来源源头。

### 4.2 raw source 层
从原件转出的可读文本层，属于 raw source 层。
正式落点为：
- `raw-sources/`

### 4.3 source record 层
对来源的记录、索引和挂接信息，属于 Source Record 层。
正式落点为：
- `wiki/sources/`

### 4.4 wiki 对象层
案例页、概念页、索引页、分析页属于 wiki 对象层。
正式落点包括：
- `wiki/cases/`
- `wiki/concepts/`
- `wiki/indexes/`
- `wiki/analysis/`

### 4.5 规则
ingest 的第一优先级不是写内容，而是先把输入放对层。
如果层级判断不清楚，默认先不正式写入 wiki 页面。

## 5. 当前阶段 ingest 的基本顺序
当前阶段的 ingest 应遵循以下最小顺序：

### 第一步：识别输入类型
先判断输入是：
- 新案例
- 新来源
- 新分析结果

### 第二步：识别对象归属
再判断它最终影响的是：
- Case
- Concept
- Source Record
- Raw Source
- 多个对象

### 第三步：识别写入层级
决定它应先进入：
- 原件记录
- raw source
- source record
- 候选更新说明
- 正式 wiki 页面

### 第四步：先形成候选更新
当前阶段默认原则是：
**先形成候选更新，再决定是否正式写回。**

### 第五步：再决定正式写入
只有在对象归属、目录归属和来源支撑足够清楚时，才进入正式写入。
正式 wiki 知识层写入默认应进入 write-back 决策；ingest 不应成为正式知识层写回的兜底机制。

## 6. 三类输入的最小处理标准

## 6.1 新案例的 ingest 标准
当输入是一个新案例时，当前阶段最小处理要求是：

### 必做
- 明确这是否是一个值得进入系统的正式案例对象
- 生成或准备生成 `case_id`
- 建立最小 Case 候选页或 Case 建页草稿
- 至少挂接一个来源入口
- 至少判断其可能关联的 Concept

### 暂不要求
- 一次写满全部字段
- 一次完成完整内容生产角度
- 一次完成全部 cross reference

### 当前阶段默认输出
- Case 候选更新
- 或最小可用 Case 初稿

## 6.2 新来源的 ingest 标准
当输入是一个新来源时，当前阶段最小处理要求是：

### 必做
- 记录原始出处
- 判断来源类型
- 形成 raw source 文本层或明确原件存在方式
- 建立 Source Record 或 Source Record 候选草稿
- 判断其主要关联对象：
  - Case
  - Concept
  - 或 general source

### 暂不要求
- 一次做完整摘要
- 一次做复杂可信度打分
- 一次建立全量引用系统

### 当前阶段默认输出
- Raw Source
- Source Record 候选更新或初稿

## 6.3 新分析结果的 ingest 标准
当输入是一个新的分析结果时，当前阶段最小处理要求是：

### 必做
- 判断它属于哪类分析结果：
  - 案例判断
  - 母题解释
  - 跨案例连接
  - 来源价值提炼
- 判断是否有来源支撑
- 判断它应写回哪个对象
- 优先形成候选更新，而不是直接写回正式页面

### 暂不要求
- 一次形成最终结论
- 立即进入正式 wiki 页面

### 当前阶段默认输出
- 候选更新说明
- 待写回建议
- 或 analysis 候选草稿

## 7. 当前阶段的默认输出规则
为了避免 ingest 失控，当前阶段默认输出遵循以下原则：

### 7.1 默认先输出候选更新
除非情况非常清楚，否则 ingest 默认先产出：
- 候选 Case 更新
- 候选 Concept 更新
- 候选 Source Record
- 候选关联关系
而不是直接大规模写正式页。

### 7.2 正式写入需要满足三个条件
至少满足以下三项中的前两项：
1. 对象归属清楚
2. 层级归属清楚
3. 来源支撑清楚

若不能满足，默认停留在候选层。

### 7.3 原则上不直接改 raw source
ingest 可以新增 raw source，但不应把 raw source 改写成总结页。

## 8. 当前阶段不允许的 ingest 行为
当前阶段 ingest 明确不允许以下行为：

### 8.1 把原件直接当 wiki 页面写入
例如：
- 直接把网页链接当成 Case 内容
- 直接把 PDF 原件当成 Concept 内容

### 8.2 把 raw source 当总结页使用
例如：
- 在 raw source 里加入大量主观判断
- 把分析结论直接写进 raw source 正文

### 8.3 在没有审核的情况下批量写正式知识页
例如：
- 批量建大量正式 Case 页
- 批量写大量正式 Concept 页
- 批量大改现有 wiki 页面

### 8.4 把不成熟的分析结果直接写回正式知识层
当前阶段应优先候选更新，而不是过早沉淀噪音。

## 9. 人与 Codex 的分工
### 9.1 Codex 可以做的 ingest 工作
当前阶段 Codex 可以：
- 判断输入类型
- 提出对象归属建议
- 协助生成 raw source 文本层
- 协助生成 Source Record 草稿
- 协助生成 Case / Concept 候选更新
- 整理候选写回内容

### 9.2 Codex 不应直接做的 ingest 工作
当前阶段 Codex 不应：
- 在无审核下批量写正式知识页
- 在无审核下把 ingest 当成正式知识层写回的兜底机制
- 自由决定复杂案例的最终对象归属
- 在来源支撑不足时形成正式结论
- 把非标准交易强行按标准买卖语义写入

### 9.3 人负责的关键动作
当前阶段人应优先负责：
- 判断是否值得正式进入系统
- 审核对象归属
- 审核 Concept 挂接
- 审核关键结论是否成立
- 决定是否从候选更新进入正式写回

## 10. 当前阶段 ingest 的最小记录要求
任何一次正式 ingest，至少应能回答以下问题：
1. 输入是什么
2. 它属于哪类对象
3. 它进入了哪一层
4. 它影响了哪些对象
5. 当前是候选更新还是正式写入
6. 它能回到哪个原始出处或 raw source

如果这 6 个问题答不清，就不应视为完成了一次合格 ingest。

## 11. 当前阶段的最小完成标准
一次 ingest 在当前阶段，如果满足以下条件，可视为达到最小标准：
- 输入类型已识别
- 对象归属已判断
- 层级归属已判断
- 已形成 raw source、source record、候选更新或正式对象中的至少一种产物
- 来源可追溯
- 没有污染 raw source 层
- 没有越过审核边界大规模写回

## 12. 本文档的作用
本文件用于为后续：
- 新案例进入系统
- 新来源进入系统
- 新分析结果进入系统
- Codex 协助 ingest
- review / write-back 前置判断

提供最小执行标准。
后续如果 ingest 机制变得更复杂，应在本文件基础上扩展版本，而不是绕开本标准自由实现。

## 13. 当前结论
当前阶段的 ingest 标准可以概括为：
- 先分层
- 再分对象
- 先候选
- 后写回
- 先保守吸收
- 不让来源层和知识层混写

只要这套最小纪律立住，系统就能开始“长”，而不会一开始就长歪。
