# Case 来源分层设计 v0.1

## 1. 文档定位
本文件用于定义本项目当前阶段中，Case 页面与来源层之间的信息分配原则。
它承接《系统设计文档 v1.0》《Schema 初稿 v0.1》《Case 模板文档 v1.0》《Source / Raw Source 模板文档 v1.0》以及相关执行标准。
本文件不负责定义来源对象字段本身，而是回答：
- 一个 Case 为什么可以拥有很多来源
- 这些来源应如何分层进入系统
- 为什么不能把所有信息直接堆进 Case 页面
- Case、Source Record、Raw Source 各自应承担什么信息密度

## 2. 文档作用
本文件的作用有四个：
1. 允许项目保留较高的信息收集宽度
2. 防止 Case 页面因来源堆积而失控
3. 固定来源层与知识层之间的信息密度差异
4. 为后续真实样本建设时的来源筛选提供设计依据

## 3. 当前阶段的基本判断
在本项目中，一个高价值 Case 往往需要大量来源支撑，例如：
- 网页
- 官方公告
- 年报
- 招股书
- 电话会纪要
- 视频 transcript
- 媒体深度报道
- 法律文件
- 行业分析材料

因此，**一个 Case 拥有很多来源，本身不是问题。**

真正的问题在于：
- 是否把所有来源都直接堆进 Case 页面
- 是否区分了核心来源与背景来源
- 是否把“信息收集宽度”和“页面表达密度”混为一谈

当前阶段的正式判断是：

**可以尽可能多地收集信息，但不应把所有信息平铺进 Case 页面。**

## 4. 当前阶段的四层处理原则
当前阶段，Case 相关来源应按以下四层处理。

### 4.1 第一层：原件层尽量多收
原始出处层包括但不限于：
- 原始网页
- PDF 原件
- 年报文件
- 公告原文
- 视频链接
- 法院文件
- 电话会原始材料

这一层的原则是：
**宁可先多收，不要过早删减。**

原因是：
- 后续很可能回头用到
- 早期过度删减会损失研究弹性
- 多收集并不直接污染知识层

### 4.2 第二层：Raw Source 尽量完整保留
当原件进入系统后，应尽可能转化为 Raw Source 或明确记录其原始出处。

Raw Source 的原则是：
- 以原始可读文本为主
- 尽量保持原始性
- 不承载过重主观总结
- 允许数量较多

这一层的重点是：
**多收集、多保留，但保持原始。**

### 4.3 第三层：Source Record 开始筛选和组织
Source Record 不是原件本体，而是来源的记录、索引和挂接层。

在这一层应开始回答：
- 这份来源是什么
- 它主要支持哪个 Case 或 Concept
- 它是核心证据还是背景补充
- 是否值得长期保留
- 后续是否应优先使用

这一层的作用是：
**把大量来源变成可管理、可追溯、可挂接的来源网络。**

### 4.4 第四层：Case 页面只保留高价值来源入口
Case 页面不应等于“所有来源的堆积页”。

Case 页面中的来源区域，应优先保留：
- 最关键的事实来源
- 最能支撑判断的来源
- 最值得优先查看的来源入口

Case 页面应服务于：
- 快速研究
- 内容生产
- 判断提炼
- 重要证据回溯

而不是服务于：
- 展示全部收集成果
- 平铺所有资料
- 充当来源仓库

## 5. 当前阶段的来源三分法
为了提高 Case 页面的可用性，当前阶段建议把来源理解为三类：

### 5.1 核心来源
核心来源是：
- 直接支撑关键事实或关键判断的来源
- 在正常研究或写稿中优先查看的来源
- 应优先出现在 Case 页面中的来源

典型例子包括：
- 官方公告
- 并购协议摘要
- 年报 / 10-K
- 核心访谈
- 高可信度深度报道

### 5.2 支撑来源
支撑来源是：
- 对核心判断有帮助
- 但不是第一优先级
- 更适合保留在 Source Record 层和完整来源列表中的来源

典型例子包括：
- 补充报道
- 侧面访谈
- 行业分析
- 次级数据材料

### 5.3 背景来源
背景来源是：
- 有助于理解上下文
- 但不直接构成关键交易证据
- 不应全部抬入 Case 页面主体

典型例子包括：
- 历史报道
- 一般评论
- 长视频解读
- 二手总结文章

## 6. 当前阶段为什么不能把所有来源都堆进 Case 页面
如果把所有来源都直接堆进 Case 页面，会出现以下问题：

### 6.1 页面会快速臃肿
Case 页会从研究控制台变成资料堆。

### 6.2 核心来源会被淹没
真正关键的证据与一般背景材料混在一起，研究效率下降。

### 6.3 AI 难以区分证据优先级
后续 AI 读取 Case 页面时，会把核心来源与背景来源混在一起，降低判断质量。

### 6.4 页面职责会漂移
Case 页面本应是：
- 单案例研究页
- 单案例判断页
- 内容输出基础页

而不是：
- 全部来源存档页

## 7. 当前阶段的正式设计结论
当前阶段，一个 Case 可以拥有很多来源，但不应把所有来源都平铺进页面主体。

应遵循以下设计原则：

- 收集宽度可以很大
- 来源保留可以很多
- Raw Source 可以较完整
- Source Record 负责筛选和组织
- Case 页面只保留高价值来源入口

也就是说：

**多收集，少直塞；先入来源层，再筛进 Case。**

## 8. 当前阶段与 Case 模板的关系
未来在 Case 模板中，来源区域应逐步朝以下方向演进：
- 核心来源
- 完整来源入口

但当前阶段不要求立刻把模板重做很重。
本文件先作为设计原则存在，后续再视真实样本验证结果决定是否下沉到模板正文中。

## 9. 当前阶段与执行标准的关系
这条设计原则会直接影响：
- ingest：来源先进入原件层 / Raw Source / Source Record
- write-back：Case 页面只写回高价值来源入口
- review：检查 Case 页面是否因为来源堆积而越权

因此，本文件是三份执行标准的上游设计依据之一。

## 10. 当前结论
当前阶段，Case 与来源层的关系可以概括为：

- 一个 Case 可以关联很多来源
- 信息收集宽度可以很大
- 但 Case 页面表达密度必须克制
- 来源应先进入 Raw Source 与 Source Record 层
- Case 页面应像研究控制台，而不是资料堆积页

只要这条原则立住，系统就能同时兼顾：
- 信息丰富度
- 研究效率
- 页面可读性
- 后续 AI 使用质量
