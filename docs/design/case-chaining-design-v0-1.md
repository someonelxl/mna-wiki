# Case 串联机制设计 v0.1

## 1. 文档定位
本文件用于定义本项目当前阶段中，多个相关 Case 之间应如何被串联和组织。
它承接《系统设计文档 v1.0》《索引与入口骨架设计 v0.1》《Case 与母题关联设计 v0.1》以及相关模板与执行标准。
本文件不负责定义单个 Case 的字段结构本身，而是回答：
- 当同一主体连续发生多个独立交易事件时，Wiki 应如何保持故事线连续性
- 单个 Case、相关案例字段和案例链页分别承担什么作用
- 用户如何在 Wiki 内快速看出“这个案例后面还有哪些相关资本动作”

## 2. 文档作用
本文件的作用有四个：
1. 固定“独立 Case”与“连续故事线”之间的组织关系
2. 避免把多个独立交易硬塞进一个超大 Case 页面
3. 让用户能快速总览同一主体或同一资本动作链上的多个 Case
4. 为后续主体入口页、专题页和更复杂导航预留基础

## 3. 当前阶段的基本判断
在当前 Wiki 中，Case 应优先代表：
**一次独立交易事件**

也就是说，只要满足以下条件中的多数，就应优先拆成新的 Case：
- 有新的正式交易动作
- 有新的交易结构
- 有新的宣布、签约、交割或重组节点
- 有新的核心分析问题
- 有新的来源体系
- 值得单独作为研究对象存在

因此，同一商业故事线中的多个资本动作，不应无限追加到同一个 Case 页面中。
但与此同时，Wiki 也不能因为拆分 Case，而丢失故事线连续性。

所以当前阶段需要引入：
**Case 串联机制**

## 4. 当前阶段的串联机制结构
当前阶段，多个相关 Case 的串联采用三层结构：

### 4.1 第一层：单个 Case
单个 Case 负责记录：
- 一次独立交易事件
- 该次交易的结构、逻辑、来源和母题

Case 的职责是：
- 保持单次交易分析的独立性
- 防止页面因连续事件叠加而失控

### 4.2 第二层：相关案例字段
每个 Case 页面中保留“相关案例”字段，用于记录：
- 与该 Case 有直接连续关系的其他 Case
- 上一事件、下一事件、平行事件或强相关事件

相关案例字段的职责是：
- 提供局部跳转能力
- 让用户在单个 Case 页面内快速看到相邻或强相关的交易事件

### 4.3 第三层：Case Series / Case Chain
当多个独立 Case 明显属于同一条连续资本动作线时，应考虑建立单独的：
- Case Series
或
- Case Chain
页面

它的职责是：
- 呈现整条连续资本动作的全貌
- 按顺序组织多个独立 Case
- 说明这些 Case 为什么属于同一条链
- 帮助用户从“单次交易视角”上升到“连续资本动作视角”

## 5. 三层结构的分工
当前阶段三层结构分工如下：

### 5.1 Case
负责回答：
- 这一次交易是什么
- 为什么发生
- 交易结构是什么
- 值得研究的核心点是什么

### 5.2 相关案例
负责回答：
- 和这次交易最直接相连的其他 Case 是哪些
- 从这里下一步该跳去看什么

### 5.3 Case Series / Case Chain
负责回答：
- 这些独立 Case 共同构成了哪一条连续故事线
- 整体顺序是什么
- 这条链的共同逻辑是什么

## 6. 当前阶段为什么不能只靠“相关案例”
只保留“相关案例”字段，虽然轻量，但不够。

原因是：
- 相关案例只能提供局部连接
- 当连续资本动作超过 2 到 3 个时，用户很难仅靠局部跳转把整条故事线看清
- 相关案例字段更适合“点对点连接”，不适合“链式总览”

因此，当同一主体已发生多次独立资本动作，且这些动作之间具有明显连续性时，仅靠相关案例字段是不够的，应考虑增加 Case Series / Case Chain 页。

## 7. 当前阶段为什么不能把整条故事线都塞进一个 Case
如果把“收购 Twitter”“X 与 xAI 合并”“未来可能的进一步整合”全部塞进一个 Case，会出现以下问题：

- 时间线过长
- 交易结构混杂
- 母题过多过散
- 页面同时承担多个独立交易的分析任务
- 单次交易的核心逻辑被后续事件冲淡

这会让 Case 从“独立研究对象”退化成“人物商业史页面”。

当前 Wiki 的核心对象应是：
**独立交易事件**
而不是：
**无限展开的人物资本传记**

## 8. 何时应建立 Case Series / Case Chain
当前阶段，满足以下条件时，建议建立单独的案例链页：

### 8.1 存在两个及以上独立 Case
这些 Case 已经分别成立，不应再强塞在一个页面里。

### 8.2 这些 Case 明显属于同一条连续资本动作线
例如：
- 同一控制人连续推动的多个资本动作
- 同一平台资产的连续重组
- 同一上市路径下的连续交易事件

### 8.3 用户有明显“整体总览”需求
也就是用户会自然提出：
- 这之后还发生了什么
- 这条线总共有几步
- 这些 Case 之间是什么关系

只要这三个条件基本成立，就应考虑建立 Case Series / Case Chain 页面。

## 9. Case Series / Case Chain 页面的最小职责
当前阶段，一个案例链页至少应包含：

### 9.1 链名称
例如：
- `SERIES-001｜马斯克的 Twitter → X → xAI 资本动作链`

### 9.2 串联对象
说明这条链围绕的是：
- 哪个主体
- 哪个平台
- 哪类资产
- 哪条连续资本动作逻辑

### 9.3 串联逻辑说明
用一小段话说明：
- 为什么这些 Case 属于同一条链
- 它们共同构成了什么连续动作

### 9.4 顺序列表
至少应包括：
- 顺序
- case_id
- 案例名称
- 关系说明

### 9.5 共同母题
如果这条链共享若干关键母题，可列出：
- 共同母题
- 各步交易中这些母题如何体现

## 10. 马斯克 / Twitter / X / xAI 例子的适用方式
在这个例子里，当前阶段建议按以下方式组织：

### 10.1 独立 Case
- `CASE-001｜马斯克收购 Twitter`
- `CASE-002｜X 与 xAI 合并`
- 未来如发生新的正式资本动作，再单独建立新的 Case

### 10.2 相关案例
在 `CASE-001` 中挂：
- `CASE-002`

在 `CASE-002` 中挂：
- `CASE-001`
- 未来如有后续 Case，再继续挂接

### 10.3 案例链页
建立一页单独的 Case Series / Case Chain：
- `SERIES-001｜马斯克的 Twitter → X → xAI 资本动作链`

用来说明：
- 取得平台控制权
- 对平台进行重组
- 通过 AI 资产整合重写估值逻辑

## 11. 当前阶段与主体入口页的关系
Case Series / Case Chain 和主体入口页不是同一回事。

### 11.1 Case Series / Case Chain
按**连续资本动作**组织。

### 11.2 主体入口页
按**人物 / 公司 / 集团主体**组织。

例如：
- “马斯克相关资本动作”属于主体入口
- “Twitter → X → xAI 资本动作链”属于案例链入口

当前阶段优先级更高的是：
**Case Chain**
因为它更直接服务于“交易事件的连续性理解”。

主体入口页可以后续再做，不急于当前阶段重建。

## 12. 当前阶段的正式规则
当前阶段应采用以下规则：

### 12.1 一次独立交易事件对应一个 Case
### 12.2 同一故事线中的不同资本动作，只要构成新的独立交易事件，就应拆为新的 Case
### 12.3 每个 Case 页面都应保留“相关案例”字段
### 12.4 当多个 Case 明显构成连续资本动作链时，应考虑建立单独的 Case Series / Case Chain 页面
### 12.5 不应把整条连续资本动作线强行塞进一个超大 Case 页面

## 13. 当前阶段的正式结论
当前阶段，Case 串联机制可以概括为：

- Case 负责记录单次独立交易事件
- 相关案例负责局部跳转和相邻连接
- Case Series / Case Chain 负责呈现整条连续资本动作线
- 三层配合，既保持单次交易的分析独立性，也保留故事线连续性

## 14. 当前结论
因此，当前阶段最合理的做法不是：
- 把所有连续事件塞成一个大 Case

而是：
- 把每次独立交易拆成独立 Case
- 用相关案例字段做局部连接
- 用案例链页做整体总览

只要这套机制立住，Wiki 就能同时做到：
- 单事件研究清晰
- 连续故事线可见
- 后续新事件易于追加
