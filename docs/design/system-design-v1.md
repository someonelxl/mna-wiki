# 系统设计文档 v1.0

## 1. 文档定位
本文件用于承接《项目总需求定位 v2.0》和《第一阶段实施方案 v1.0》，把总纲中的长期方向转化为系统设计层的结构说明。
本文件回答的不是“为什么做”，而是“这个系统在结构上准备怎么长”。
本文件属于设计层文档，不替代总纲，不替代 Rulebook，不替代模板文档。

## 2. 设计目标
本系统设计应满足以下目标：
1. 严格对齐 Karpathy Wiki 的基本结构：raw sources / wiki / rulebook
2. 以并购案例为核心，但不把系统收窄为单案例容器
3. 允许来源、概念与案例共同驱动知识增长
4. 支持长期沉淀、连接、发现与内容输出
5. 便于 Git/GitHub、VS Code、Obsidian、Codex 协同
6. 可持续扩展，不因第一阶段的简化而破坏长期结构

## 3. 系统的三层基本结构
### 3.1 Raw Sources 层
这是原始资料层，是 source of truth 的文本化承载层。
它负责承接：
- 网页转 Markdown 后的原始文本
- PDF 提取后的文本层
- 视频 transcript 文本层
- 书籍、研报、文献、报告等可读原始文本

这一层的核心职责是：
- 保存原始可读资料
- 为 ingest 提供输入
- 为 wiki 提供证据来源
- 尽量保持原始性与可追溯性

### 3.2 Wiki 层
这是整理后的知识层。
它不是原始资料仓库，而是系统中可直接浏览、连接、复用的知识页面层。
它负责承载：
- Case 页面
- Concept 页面
- Index 页面
- 未来可能存在的 analysis / summary 页面

这一层的核心职责是：
- 组织知识
- 建立连接
- 承接 write-back
- 服务研究与内容生产

### 3.3 Rulebook / Schema 层
这是规则层。
它负责定义：
- 对象边界
- 命名规则
- 写入边界
- ingest / write-back / review 的约束
- 目录结构与页面类型的解释方式

这一层的核心职责是：
- 固定系统秩序
- 降低人和 AI 的自由歧义
- 保证系统长期可控

## 4. 核心对象设计
### 4.1 Case
Case 是系统最重要的核心对象。
它代表具体并购案例，是最主要的研究对象、组织单位和内容输出对象。
Case 的设计意义在于：
- 承接单案例研究
- 吸收与该案例相关的原始资料
- 连接相关 Concept
- 连接相关 Case
- 为内容生产提供最直接的结构化基础

### 4.2 Concept
Concept 是系统中的正式连接层。
它不是普通标签，而是母题、分析框架与案例之间的中间结构。
Concept 的设计意义在于：
- 连接多个 Case
- 连接多个 Source
- 承接已有母题库
- 允许系统从来源中长出新母题和新分析视角
- 支持从概念出发组织案例和内容方向

### 4.3 Source
Source 是系统的来源对象，但应区分为两层理解：
1. raw source 文本层
2. source record 记录层

Source 的设计意义在于：
- 为系统提供原始资料输入
- 为判断提供可追溯依据
- 为 Concept 与 Case 的生长提供来源支持

## 5. 核心关系设计
### 5.1 Case 与 Source
- 一个 Case 可以关联多个 Source
- 一个 Source 可以只属于一个 Case，也可以同时影响多个 Case
- Case 不应被理解为“包含一切 source 的容器”，而是 source 的主要吸收对象之一

### 5.2 Concept 与 Case
- 一个 Concept 可以连接多个 Case
- 一个 Case 也可以关联多个 Concept
- Concept 不只是给 Case 打标签，而是提供分析框架、母题组织与跨案例连接

### 5.3 Source 与 Concept
- 一个 Source 可以强化已有 Concept
- 一个 Source 也可以推动新 Concept 的形成
- 书籍、研报、文献、报告等 general source 更应优先通过 Concept 连接到多个 Case

### 5.4 Case、Concept、Source 不是串行关系
系统不应被理解成单一路径。
更准确的理解应是：
- Case 是核心对象
- Concept 是正式连接层
- Source 是知识增长来源层
三者持续相互作用，形成知识网络

## 6. 知识增长方向设计
### 6.1 单案例增长
系统允许围绕单个案例吸收来源、形成判断、建立链接。
这是最常见的增长方式。

### 6.2 概念驱动增长
系统允许从已有母题或新概念出发，去组织相关案例。
这支持：
- 母题驱动的内容策划
- 案例群的对照分析
- 母题体系的持续扩展

### 6.3 来源驱动增长
系统允许书籍、研报、文献、报告等 general source 进入系统后，不直接归属于某个 Case，而是先参与 Concept 生长，再影响多个 Case。
这支持：
- 新母题发现
- 新分析角度形成
- 对既有案例的重读和重写

## 7. 两类入口的设计边界
### 7.1 信息流入入口
信息流入入口解决的是“系统如何长大”。
它包括：
- 新案例进入
- 新来源进入
- 新分析结果写回
- review 后的修正进入

这一层最终服务于：
- raw sources 增长
- wiki 更新
- 连接扩展

### 7.2 用户使用入口
用户使用入口解决的是“人如何进入和使用系统”。
它未来至少应支持：
- 从 Case 进入
- 从 Concept 进入
- 从索引与搜索进入

这一层现在只定义方向，不做交互细节设计。

### 7.3 当前设计原则
- Case 是主组织单位和主要使用场景
- 但系统不应被限制为只有单一入口
- Concept 和 Source 也能成为知识增长的起点
- 具体首页、目录、搜索与浏览设计，后续放在独立设计文档中展开

## 8. 项目目录与系统层的对应关系
### 8.1 `docs/`
承载：
- 总纲文档
- 实施文档
- 设计文档

### 8.2 `rules/`
承载：
- Rulebook
- Schema
- 标准文档

### 8.3 `templates/`
承载：
- Case / Concept / Source 模板
- 其他结构模板

### 8.4 `raw-sources/`
承载：
- raw source 文本层
- 按 cases / general / inbox 组织的来源输入

### 8.5 `wiki/`
承载：
- cases
- concepts
- indexes
- analysis

### 8.6 `scripts/`
承载：
- ingest 相关脚本
- source 处理脚本
- indexing 相关脚本
- 通用 utilities
这是系统接线与轻量自动化层，不是知识层本身。

## 9. Obsidian、VS Code、Codex 的结构角色
### 9.1 Obsidian
Obsidian 是知识浏览、页面维护与结果审阅界面。
它更适合：
- 阅读 wiki 页面
- 维护结构化知识页面
- 查看连接结构
- 承接日常知识工作

### 9.2 VS Code
VS Code 是项目开发与结构施工后台。
它更适合：
- 维护目录结构
- 编辑规则与模板
- 管理脚本与配置
- 让 Codex 在正式项目目录中工作

### 9.3 Codex
Codex 是受约束的施工队。
它在这一系统中的设计角色是：
- 先读总纲与规则
- 再参与设计与模板落地
- 执行轻量接线与结构施工
- 不越过总纲和规则自由决定系统方向

## 10. 当前阶段不在本设计文档中展开的内容
为了保持设计层边界清晰，本文件暂不展开：
- Case 模板的具体栏目
- Concept 模板的具体栏目
- Source / Raw Source 模板的具体字段
- Rulebook 的具体条文
- 首页与搜索界面的详细交互
- 自动化流程和脚本实现细节

这些内容应分别下沉到：
- 模板文档
- Rulebook / Schema
- 后续交互设计文档
- 实现文档

## 11. 当前设计结论
本系统的设计核心可以概括为：
1. 以 Case 为核心对象
2. 以 Concept 为正式连接层
3. 以 Source 为知识增长来源层
4. 以 raw sources / wiki / rulebook 为长期结构骨架
5. 允许系统同时支持沉淀、连接、发现与输出
6. 先搭结构，再长内容，再做自动化

## 12. 本文档的使用方式
本文件是当前阶段的系统结构说明文档。
以后所有模板、规则、入口设计和轻量实现，都应以本文件描述的结构边界为基础。
如果未来系统结构发生调整，应优先更新本文件，而不是让局部实现脱离整体结构。
