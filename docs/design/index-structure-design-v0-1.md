# 索引与入口骨架设计 v0.1

## 1. 文档定位
本文件用于定义本项目当前阶段的索引与入口骨架。
它承接《项目总需求定位 v2.0》《系统设计文档 v1.0》《Rulebook 初稿 v0.1》《Schema 初稿 v0.1》以及相关模板与执行标准。
本文件不负责定义对象结构本身，也不负责定义交互式搜索或 UI 实现，而是回答：
- 当前阶段系统应有哪些索引页
- 这些索引页各自承担什么作用
- 用户应如何从索引进入 Case、Concept、Source Record
- 当前阶段入口层应先搭到什么程度

## 2. 文档作用
本文件的作用有四个：
1. 定义当前阶段最小可用的入口体系
2. 固定 `wiki/indexes/` 的首版结构
3. 帮助人和 LLM 从“散对象”进入“可浏览系统”
4. 为后续搜索、首页和更复杂导航预留基础

## 3. 当前阶段的设计原则
### 3.1 入口先轻，不做重 UI
当前阶段只做结构入口，不做完整交互界面，不做复杂搜索产品，不做过重首页。

### 3.2 入口服务于使用，而不是装饰
索引页的目标不是好看，而是让用户能：
- 找到对象
- 看清对象分组
- 理解对象之间的关系
- 更快进入研究和内容生产

### 3.3 Case 仍然是主使用入口
虽然系统支持从 Concept 和 Source 出发增长，但当前阶段用户最常见的进入方式仍然应是：
- 从 Case 进入

### 3.4 Concept 是第二入口
Concept 不是附属标签，而是正式连接层。
因此它应拥有独立、清晰的入口。

### 3.5 Source 入口先轻量
当前阶段 Source Record 入口应存在，但不需要做得过重。
它的主要价值是：
- 可回溯
- 可挂接
- 可作为研究支撑入口
而不是成为主浏览首页。

## 4. 当前阶段的索引对象范围
当前阶段正式纳入索引骨架的对象有三类：
- Case
- Concept
- Source Record

当前阶段暂不把以下对象作为独立入口重点：
- Raw Source
- Analysis
- Summary
- Decision Log

这些对象后续可扩展，但当前阶段不应让入口层过重。

## 5. 当前阶段应建立的索引页
当前阶段建议建立以下 3 类正式索引页：

### 5.1 Case Index
作用：
- 作为 Case 的主入口页
- 列出当前正式纳入系统的案例对象
- 支持后续按行业、交易结构、地区、状态等维度扩展

### 5.2 Concept Index
作用：
- 作为 Concept 的主入口页
- 帮助从母题和分析框架进入案例网络
- 支持后续按母题类型、优先级、状态等维度扩展

### 5.3 Source Index
作用：
- 作为 Source Record 的轻量入口页
- 帮助查看当前已纳入系统的正式来源记录
- 主要服务于可追溯性和研究支撑，不追求成为主入口

## 6. 当前阶段不建议单独建立的入口页
以下入口当前阶段暂不建议正式建立为重点导航页：
- Raw Source Index
- Analysis Index
- Summary Index

### 原因
#### Raw Source
raw source 属于原始可读层，不应作为日常浏览入口重点。

#### Analysis / Summary
当前阶段还未进入大规模分析页沉淀阶段，入口过早建立会让结构过重。

## 7. `wiki/indexes/` 的首版结构建议
当前阶段建议 `wiki/indexes/` 下至少存在以下 3 个页面：
- `case-index.md`
- `concept-index.md`
- `source-index.md`

### 命名原则
文件名继续采用 `kebab-case`。
页面标题保持人类可读。

## 8. 各索引页的最小职责

## 8.1 Case Index 的最小职责
Case Index 当前阶段至少应承担：
1. 列出当前正式 Case 页面
2. 提供最基本的案例识别信息
3. 帮助快速进入单案例页面
4. 为后续分组和筛选预留位置

### 当前阶段建议包含的信息
- `case_id`
- `case_name`
- `transaction_structure`
- `industry`
- `region`
- `status`
- 一句话定义（可选，轻量）

### 当前阶段不要求
- 复杂筛选器
- 全量标签系统
- 自动排序系统
- 重交互搜索

## 8.2 Concept Index 的最小职责
Concept Index 当前阶段至少应承担：
1. 列出当前正式 Concept 页面
2. 提供母题识别信息
3. 帮助从母题进入案例网络
4. 为后续分类和分层预留位置

### 当前阶段建议包含的信息
- `concept_id`
- `concept_name`
- `concept_type`
- `status`
- 核心逻辑一句话（可选，轻量）

### 当前阶段不要求
- 复杂母题树
- 多层级知识图谱
- 自动关系图渲染

## 8.3 Source Index 的最小职责
Source Index 当前阶段至少应承担：
1. 列出当前正式 Source Record 页面
2. 提供来源识别信息
3. 帮助从知识层回到来源记录层
4. 为后续来源池管理预留位置

### 当前阶段建议包含的信息
- `source_id`
- `title`
- `source_type`
- `author_or_publisher`
- `published_at`
- 主要关联对象（可选，轻量）

### 当前阶段不要求
- 复杂来源可信度评分
- 自动来源聚类
- Raw Source 全量浏览入口

## 9. 首入口的组织方式
当前阶段，系统首入口建议采用“索引并列入口”而不是“单首页总控”。

也就是说，当前阶段更推荐：
- `Case Index`
- `Concept Index`
- `Source Index`

三者并列存在，而不是一开始就设计一个很重的首页。

### 原因
1. 当前阶段对象数量还不大
2. 规则层和模板层刚刚稳定
3. 入口层应先轻量验证，不宜过早做重总控页

## 10. 用户进入系统的三条主路径
当前阶段，用户使用系统的主路径应为：

### 路径一：从 Case 进入
最常见的路径。
适合：
- 单案例研究
- 视频选题
- 交易逻辑分析

### 路径二：从 Concept 进入
第二重要路径。
适合：
- 从母题组织案例
- 从方法论寻找内容角度
- 做跨案例对照

### 路径三：从 Source 进入
辅助路径。
适合：
- 回溯来源
- 核实判断
- 从书籍、研报、文献出发找到相关对象

## 11. 当前阶段入口层与对象层的关系
### 11.1 索引页不是对象替代品
Index 页只负责组织入口，不替代 Case / Concept / Source Record 本身。

### 11.2 索引页应尽量轻
Index 页不应重复对象全文，只应提供最小识别信息与跳转入口。

### 11.3 索引页是浏览层，不是规则层
它们不负责定义对象结构，只负责帮助使用。

## 12. Source Index 的边界与单案例来源组织

### 12.1 问题背景
当前阶段，Source Index 被定义为 Source Record 的轻量全局入口。
但在实际使用中，如果某一个 Case 关联大量来源，例如 20 个、30 个甚至更多，单纯依赖 Source Index 平铺来源记录，会出现以下问题：
- 页面会快速变长
- 从单案例角度查找关键来源会变得低效
- 来源的重要性层级不清楚
- 用户难以快速判断哪些来源是真正支撑判断的核心来源

### 12.2 当前阶段的结论
Source Index 应继续保留，但必须明确其职责边界：
- Source Index 是**全局来源入口**
- Source Index 不是**单案例来源精细管理页**
- Source Index 主要服务于：
  - 查看系统中有哪些正式 Source Record
  - 从知识层回到来源层
  - 进行全局来源浏览与回溯
- Source Index 不应承担：
  - 单案例内部全部来源的精细整理
  - 单案例来源优先级判断
  - 单案例来源分组管理的主职责

### 12.3 单案例来源过多时的处理原则
当某一个 Case 关联大量来源时，复杂度应优先在 Case 页面内部消化，而不是压到 Source Index 上。

当前阶段建议采用以下原则：

#### 12.3.1 Case 页面区分“核心来源”与“完整来源”
Case 页面中的来源区域，后续应逐步区分：
- 核心来源：真正支撑关键判断的主要来源
- 完整来源：该案例全部正式纳入的来源记录

#### 12.3.2 优先展示核心来源
在正常阅读和研究场景中，应优先看到核心来源，而不是一开始平铺全部来源。

#### 12.3.3 完整来源可后置
完整来源列表可以保留，但应后置或分组呈现，避免破坏页面可读性。

#### 12.3.4 来源应允许按类型分组
当来源数量较多时，后续可考虑按以下类型分组：
- 官方文件 / filing
- 公司公告 / 年报
- 媒体报道
- 访谈 / transcript
- 第三方分析
- 视频材料

### 12.4 对 Source Index 的影响
这一修正意见不要求当前阶段重做 Source Index。
当前阶段，Source Index 仍维持轻量入口角色即可。

但后续在继续扩展入口层时，应注意：
- Source Index 不应被做成“单案例来源工作台”
- Source Index 的字段和展示方式，应更偏全局来源识别
- 单案例来源组织能力，应主要回归 Case 页面内部结构

### 12.5 当前阶段的设计判断
因此，当前阶段可以接受 Source Index 随着来源增多而逐步变长，但不应因此误判为 Source Index 设计失败。
真正需要补强的，不是 Source Index 本身，而是：
- Case 页面中的来源组织方式
- 核心来源与完整来源的区分
- 来源分组和优先级呈现方式

### 12.6 当前结论
当前阶段的正式结论是：
- Source Index 是全局来源入口
- Source Index 不是单案例来源精细管理页
- 单案例来源数量过多时，复杂度应优先在 Case 页面内部处理
- 后续应在 Case 模板或 Case 页面设计中逐步引入“核心来源 / 完整来源 / 来源分组”的组织方式

## 13. 当前阶段的最小完成标准
如果以下条件满足，可视为入口层达到当前阶段最小可用状态：
1. `wiki/indexes/` 下有 Case / Concept / Source 三个索引页
2. 三类索引页各自职责清楚
3. 用户可从索引进入对象页
4. 索引页不与对象页职责混写
5. 入口层没有过早做重

## 14. 当前阶段不在本文件中展开的内容
当前阶段暂不在本文件中展开：
- 全项目首页
- 搜索设计
- 筛选器设计
- graph / 可视化导航
- 自动索引脚本
- 动态索引生成逻辑

这些内容后续如需要，应下沉到更具体的设计或实现文档中。

## 15. 本文档的作用
本文件的作用，是在当前阶段先把系统的入口骨架立起来。
以后：
- Index 模板设计
- Index 页面创建
- 轻量导航整理
- 后续搜索和入口升级

都应以本文件为基础。

## 16. 当前结论
当前阶段的入口骨架可以概括为：
- 先立三类索引页
- 以 Case 为主入口
- 以 Concept 为第二入口
- 以 Source 为轻量辅助入口
- 不急着做重首页和重搜索
- 不把 Source Index 做成单案例来源工作台
- 单案例来源复杂度优先在 Case 页面内部消化

只要这三层入口先立住，系统就已经从“有对象”进入“可浏览、可进入、可组织”的状态。
