# 第一阶段实施方案 v1.0

## 1. 文档目的
本文件用于承接《项目总需求定位 v2.0》，将总纲中的长期方向转化为第一阶段可执行的推进计划。
本文件属于阶段性实施文档，负责说明当前阶段的目标、边界、任务顺序与交付物，不替代总需求定位文档，也不替代后续的系统设计文档、规则文档和模板文档。

## 2. 第一阶段的核心目标
本阶段不追求系统做完，而追求把项目从“总纲明确”推进到“可以持续施工”。

本阶段的核心目标有四个：
1. 建立正式、稳定、可持续扩建的项目工作环境
2. 明确第一版系统设计骨架
3. 建立规则层与模板层的初始文档体系
4. 让 Codex 能在明确约束下开始参与施工

## 3. 第一阶段的边界
### 3.1 本阶段要做什么
本阶段主要做：
- 稳定项目环境
- 完成实施层与设计层文档
- 建立 Rulebook / Schema 初稿
- 建立 Case / Concept / Source 模板初稿
- 形成后续可持续施工的基础秩序

### 3.2 本阶段暂不做什么
本阶段先不做：
- 大规模自动化
- 复杂脚本系统
- 完整 source 处理流水线
- 完整首页 / 搜索 / 交互设计落地
- 大量正式案例录入
- 大规模知识写回和批量 ingest

本阶段的重点是先搭施工体系，而不是先铺满内容。

## 5. 第一阶段的任务顺序
### 第一步：巩固环境层
确认以下基础设施已成立并可用：
- 项目根目录
- Git 仓库
- GitHub 远程仓库
- README
- `.gitignore`

### 第二步：建立实施层文档
完成本文件，作为当前阶段的施工起点。

### 第三步：建立系统设计文档
把总纲中已经确定的长期结构，下沉为设计层结构，包括：
- Case / Concept / Source 的关系
- 两类入口的设计边界
- raw sources / wiki / rules / templates 的关系
- 项目目录结构与知识流方向

### 第四步：建立规则层文档
开始起草 Schema / Rulebook，优先固定最核心的规则：
- 命名规则
- 对象边界
- 写入边界
- ingest / write-back / review 的原则边界

### 第五步：建立模板层文档
完成三类核心模板的初稿：
- Case
- Concept
- Source / Raw Source

### 第六步：在明确约束下让 Codex 参与施工
在总纲、命名规范、实施方案、设计文档和规则初稿都已存在的前提下，让 Codex 开始参与：
- 文档结构整理
- 模板文件落地
- 规则文件骨架搭建
- 轻量 glue code 准备

## 6. Codex 在第一阶段的角色
### 6.1 应承担的角色
Codex 在第一阶段的角色应是：
- 阅读总纲与规则
- 协助搭建设计层文档
- 协助生成模板与规则骨架
- 执行轻量、受约束的结构施工

### 6.2 不应承担的角色
第一阶段不应让 Codex：
- 自由决定系统方向
- 在没有规则约束时大规模重构结构
- 在没有审核的情况下批量生成正式知识内容
- 越过总纲与规则文档自由发挥

第一阶段的 Codex 是受约束施工队，不是自由总设计师。

## 7. 第一阶段完成的判断标准
当以下条件基本满足时，可以认为第一阶段基本完成：
1. 总纲文档稳定
2. 项目环境稳定
3. 系统设计文档初版完成
4. Rulebook / Schema 初版完成
5. Case / Concept / Source 模板初版完成
6. Codex 已能在明确约束下参与结构施工
7. 项目从“准备阶段”进入“可持续建设阶段”

## 8. 本文档的使用方式
本文件是当前阶段的施工图。
后续如果阶段目标、任务顺序或交付物发生变化，应优先更新本文件，而不是直接改动总纲文档。
本文件会随阶段推进迭代版本，但始终受《项目总需求定位 v2.0》约束。

## 9. 当前阶段的总判断
第一阶段的核心任务不是做出完整系统，而是把：
- 方向
- 环境
- 设计
- 规则
- 模板
- Codex 施工边界

这六件事先立起来。

只要这六件事立住，项目就具备了持续建设的基础。
