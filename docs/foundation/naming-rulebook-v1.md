# 命名规范总纲 v1.0

## **1. 文档定位**

这是一份用于长期复用的命名规范总纲。
它的作用不是规定某一个具体项目的全部命名细节，而是提供一套稳定、清晰、可持续复用的命名原则，使人或 LLM 在面对一个新的项目、一个新的文件夹、一个新的文件或一个新的知识对象时，都能快速判断：

- 应该用什么命名风格
- 应该优先体现什么信息
- 哪些命名方式应当避免

这份文档属于**原则层与指导层**，不负责承载阶段性、项目专属或实施层的具体命名清单。

## **2. 命名规范的总原则**

### **原则一：命名首先服务于长期维护**
命名不是装饰，而是系统秩序的一部分。

一个名字应首先保证：

- 长期可理解
- 长期可复用
- 长期可搜索
- 长期可排序
- 长期不易失效
### **原则二：命名首先服务于结构，而不是服务于情绪**
命名应优先表达对象在系统中的位置、类型和作用，而不是表达一时的感觉、口号或临时判断。
### **原则三：命名应优先服务于人和 LLM 的共同理解**
命名体系既要让人类快速理解，也要让 LLM 易于识别、分类、比较和调用。

因此，命名应尽量：

- 清楚
- 稳定
- 一致
- 少歧义
### **原则四：命名必须分层，而不是全系统只用一种风格**
不同层级、不同类型的对象，应采用不同但一致的命名方式。

例如：

- 项目与文件系统层
- 知识对象层
- 页面显示层
- 规则文档层

这些层不应混用同一套命名逻辑。
### **原则五：命名应优先选择成熟、通行、工程化的风格**
当存在成熟、稳定、被广泛验证的命名风格时，应优先采用它们，而不是自创风格。

---

## **3. 命名层级的基本划分**

### **3.1 文件系统层**
文件系统层包括：

- 项目根目录
- repo 名称
- 文件夹名称
- 文件名
- 脚本名
- 配置文件名

这一层的命名首先服务于：

- 路径稳定
- Git/GitHub 兼容
- Terminal 友好
- 跨工具使用一致
### **3.2 知识对象层**
知识对象层包括：

- 案例对象
- 来源对象
- 母题 / Concept 对象
- 决策记录对象
- 其他可编号的知识实体

这一层的命名首先服务于：

- 稳定引用
- 可排序
- 可追踪
- 可长期扩展
### **3.3 显示层**
显示层包括：

- 页面标题
- 文档标题
- README 标题
- 对外展示名称

这一层的命名首先服务于：

- 人类阅读
- 快速理解
- 表达清晰
### **3.4 规则层**
规则层包括：

- Rulebook
- Schema
- 标准文档
- 规则说明文件

这一层的命名首先服务于：

- 一眼识别其规则属性
- 与普通文档、模板文档区分开

---

## **4. 各层推荐采用的命名风格**

## **4.1 文件系统层：统一采用 kebab-case**

文件系统层推荐统一使用：

**kebab-case**

其特点是：

- 全小写
- 单词之间用 - 连接

推荐原因：

- 兼容 GitHub、Terminal、URL、脚本路径
- 清晰、稳定、成熟
- 人和 LLM 都容易识别

因此，新的：

- repo 名
- 项目根目录名
- 文件夹名
- 模板文件名
- 普通文件名
- 脚本名

原则上都应优先使用 kebab-case。

---

## **4.2 知识对象层：统一采用“前缀 + 编号”体系**

知识对象层推荐统一使用：

**前缀 + 编号**

其作用是：

- 稳定引用
- 保持排序
- 支持扩展
- 降低歧义

适合这种方式的对象包括：

- Case
- Source
- Concept / 母题
- 决策记录
- 其他核心知识实体

命名时应优先保证：

- 前缀语义稳定
- 编号连续或可持续扩展
- 不因阶段变化而轻易重命名

---

## **4.3 显示层：优先采用自然语言标题**

显示层不应强行工程化。

它应优先服务于：

- 人类阅读
- 表达清楚
- 减少理解成本

因此，显示层标题应优先使用自然语言标题。

显示层允许：

- 中文
- 中英文混合
- 更完整的表达

但仍应保持：

- 克制
- 明确
- 不冗长
- 不口号化

---

## **4.4 规则层：统一使用可识别前缀**

规则层文件应当一眼可识别。

因此推荐统一采用前缀体系，例如：

- rulebook-
- schema-
- standard-

其核心原则不是死记某一个前缀，而是：

**规则类文档必须在名称上与普通文档、模板文档、知识页面明确区分。**

---

## **5. 命名时应优先体现的信息顺序**

无论命名哪个对象，都应优先考虑以下信息顺序：
### **第一优先：它属于哪一层**
先判断这个对象属于：

- 文件系统层
- 知识对象层
- 显示层
- 规则层
### **第二优先：它是什么类型**
例如：

- case
- concept
- source
- template
- rulebook
- schema
- script
### **第三优先：它的主题或作用是什么**
例如：

- ingest
- naming
- review
- source-record
- case-template
### **第四优先：是否需要稳定编号**
如果该对象未来需要长期追踪、排序或引用，应优先采用编号体系，而不是纯文字自由命名。

---

## **6. 命名时应避免的做法**

### **6.1 避免混用多种风格**
同一层级内，不应同时混用：

- kebab-case
- snake_case
- camelCase
- 中文随意命名
- 拼音随意命名
### **6.2 避免使用情绪化或阶段性后缀**
例如：

- new
- final
- final-final
- latest
- 最新版
- test
- 临时

这些词通常会快速失效，不适合作为长期命名的一部分。
### **6.3 避免把过多语义塞进文件名**
命名应体现对象身份和主题，但不应把完整说明句写进名字。
### **6.4 避免为了“好看”牺牲结构清晰度**
命名的首要目标是稳定、清楚、可维护，而不是装饰性。
### **6.5 避免使用只对当前自己有意义、但未来难以理解的缩写**
缩写只有在已经稳定、明确、可长期复用时，才适合进入正式命名体系。

---

## **7. 命名决策时的标准流程**

以后无论是人还是 LLM，在给新的项目、新的文件夹、新的文件或新的对象命名时，都应遵循以下流程：
### **第一步：先判断对象层级**
先判断它属于哪一层：

- 文件系统层
- 知识对象层
- 显示层
- 规则层
### **第二步：再判断它的类型**
例如它是：

- 页面
- 文件夹
- 模板
- 脚本
- 规则文档
- 知识对象
### **第三步：选择该层推荐的命名风格**
而不是自由发挥。
### **第四步：检查名字是否长期稳定**
应判断：

- 以后是否容易失效
- 是否会导致歧义
- 是否容易与其他对象冲突
- 是否可持续扩展
### **第五步：最后再看是否足够简洁、清楚、可搜索**

---

## **8. 这份总纲对人和 LLM 的约束**

### **对人**
在新增任何项目、目录、文件、知识对象前，应先对照本总纲判断：

- 它属于哪一层
- 应采用哪种命名风格
- 是否需要编号
- 是否存在更成熟、更一致的命名方式
### **对 LLM**
在为系统生成新的项目名、目录名、文件名、对象名时，应默认遵守：

- 文件系统层优先 kebab-case
- 知识对象层优先前缀 + 编号
- 显示层优先自然语言标题
- 规则层优先显式前缀区分

若无充分理由，不应自行发明与本总纲冲突的新命名风格。

---

## **9. 这份文档的作用**

这份文档的作用，是为未来所有项目与文件命名提供长期稳定的指导原则。

它不是某一个项目的具体命名清单，而是一个可反复复用的命名标准总纲。

未来如果有新的项目、新的目录、新的模板、新的规则文件或新的知识对象，应优先以本总纲作为命名判断依据。

如果未来出现新的命名场景，也应优先扩展这份总纲，而不是随意在局部自创规则。
